# 仕様書: Image-3D — 画像から3Dプリントデータ生成アプリ

## 1. 概要

アップロードされた画像(単一イメージ、またはキャラクターシート等)を元に、
3Dプリンターで使用できるデータ(STL / 3MF / GLB / OBJ)を生成するWebアプリケーション。
生成した3Dモデルはアプリ内の3Dビューアでインタラクティブに確認できる。

## 2. ターゲット環境

| 項目 | 内容 |
|---|---|
| 実行環境 | ローカルマシン(Linux, NVIDIA RTX PRO 6000 Blackwell 96GB VRAM) |
| バックエンド | Python 3.12 / FastAPI / Uvicorn |
| 3D生成モデル | Hunyuan3D-2 (shape generation) を第一候補。プラガブル設計で差し替え可能 |
| メッシュ処理 | trimesh (+ pymeshlab はオプション) |
| フロントエンド | HTML + ES Modules + Three.js(ビルド不要、FastAPIが静的配信) |
| ブラウザ | Chrome / Firefox / Edge 最新版(WebGL2必須) |

## 3. 機能要件

### 3.1 画像アップロード (FR-1)
- PNG / JPEG / WebP をドラッグ&ドロップまたはファイル選択でアップロードできる。
- 最大ファイルサイズ: 20MB。
- キャラクターシート(複数ビューが1枚に並んだ画像)の場合、正面ビューを主入力として使用する
  (マルチビュー入力対応は将来拡張)。
- アップロード画像のプレビューを表示する。

### 3.2 背景除去 (FR-2)
- 生成精度向上のため、アップロード画像から背景を自動除去できる(rembg使用)。
- ON/OFF切り替え可能(デフォルトON)。除去結果をプレビュー表示する。

### 3.3 3Dモデル生成 (FR-3)
- 画像1枚から3Dメッシュを生成する(Image-to-3D)。
- 生成はジョブとして非同期実行し、進捗状態(queued / preprocessing / generating /
  postprocessing / completed / failed)をポーリングAPIで取得できる。
- 生成パラメータ:
  - `steps`: 拡散ステップ数(デフォルト30)
  - `guidance_scale`: ガイダンススケール(デフォルト5.5)
  - `octree_resolution`: メッシュ解像度(256 / 384 / 512、デフォルト384)
  - `seed`: 乱数シード(省略時ランダム)
- ジェネレータはプラガブル(`Generator`インターフェース):
  - `hunyuan3d`: Hunyuan3D-2 shape pipeline(本番用)
  - `pixal3d`: Pixal3D(TencentARC、TRELLIS.2基盤、重みまでMITライセンス)。単一画像
    からPBRテクスチャ付きメッシュを生成し、テクスチャは頂点カラーとして活用する。
    専用venv(.venv-pixal3d、requirements-pixal3d.txt)での起動が前提で、
    IMAGE3D_GENERATOR=pixal3d の明示指定のみ(autoでは解決しない)。
    マルチビュー入力(FR-9)非対応。パラメータは steps/seed のみ接続。
  - `mock`: 入力画像に依らず既知メッシュ(例: Utahティーポット/スザンヌ相当)を返す
    開発・テスト用ジェネレータ。モデル未ダウンロード環境でもアプリ全体が動作すること。

### 3.4 3Dプリント向けメッシュ後処理 (FR-4)
生成メッシュに対し以下を自動適用する(trimesh):
- 非多様体・重複面・浮遊小部品(全体積の1%未満の連結成分)の除去
- 穴埋め(watertight化の試行)と watertight 判定結果のレポート
- 法線の統一
- 指定サイズへのスケーリング(目標高さ mm 指定、デフォルト100mm)
- ポリゴン数の簡略化(目標面数指定可、デフォルト200,000面上限)

### 3.5 3Dビューア (FR-5)
- 生成完了したモデルをアプリ内でThree.jsによりレンダリング表示する。
- 操作: 回転(左ドラッグ)/ パン(右ドラッグ)/ ズーム(ホイール)= OrbitControls。
- 表示モード: シェーディング / ワイヤーフレーム切替。
- グリッド床とビルドプレートサイズ目安(220×220mm相当)の表示。
- モデル情報表示: 頂点数・面数・バウンディングボックス寸法(mm)・watertight判定・推定体積(cm³)。

### 3.6 エクスポート (FR-6)
- STL(バイナリ)/ 3MF / GLB / OBJ 形式でダウンロードできる。
- STL/3MFは後処理済み(プリント向け)メッシュを出力する。

### 3.7 4色カラープリント対応 (FR-8)
マルチフィラメント方式のカラー3Dプリンタ(Bambu Lab AMS、Prusa MMU等、最大4色)向けの出力に対応する。

- **カラーモード** (`color_mode`): `none`(単色、デフォルト)/ `color4`(4色カラー)。
- **色付けの方式**: 入力画像(背景除去後)をメッシュ正面軸に沿って直交投影し、
  頂点カラーを取得する(テクスチャ生成AIは使用しない)。正面画像は正面側の頂点に
  のみ投影し、背面画像がある場合は背面側の頂点に背面画像を投影する。背面画像が
  無い場合、背面側と側面/上下の曖昧な頂点はベース色にする。
- **パレット量子化**: 頂点カラーをk-meansで `n_colors`(2〜4、デフォルト4)色に量子化する。
  量子化後のパレット(HEX)と色ごとの面数比率をジョブ`stats.palette`として返す。
- **エクスポート**:
  - 3MF: 色ごとにメッシュを分割し、1ファイル内に最大4オブジェクト
    (名前 `color_1`〜`color_4`、表示色付き)として出力。スライサー側で
    オブジェクトごとにフィラメントを割り当てて印刷する。
  - GLB: 頂点カラー付きで出力し、ビューアでカラー表示する。
  - STL/OBJ: 形状のみ(色情報なし、従来通り)。
- **ビューア**: カラーモード時は頂点カラーを表示。モデル情報バーにパレット
  (色チップ+面数比率)を表示する。
- 色分割された各サブメッシュは後処理(FR-4)適用後のメッシュから生成するため、
  合計形状は単色出力と一致する。サブメッシュ単体はwatertightとは限らない点を
  UIに注記する(積層方式のマルチカラー印刷では通常問題にならない)。

### 3.8 キャラクターシート/マルチビュー入力 (FR-9)【Phase 3】
- 複数ビュー画像(正面必須+背面/左側面/右側面の任意組合せ)をビューラベル付きで
  アップロードでき、複数ビュー時は Hunyuan3D-2mv(multiview shapeモデル)で生成する。
  単一画像時は従来の hunyuan3d-dit-v2-0 を使用する。
- **キャラクターシート自動分割**: 1枚のシート画像から被写体パネルを自動検出
  (背景/アルファの連結成分解析)し、検出パネルをUI上でユーザーが各ビューに
  割り当てて生成に使用できる。自動割当の初期推定は左からの並び順ヒューリスティクス
  (正面→側面→背面の順を仮定)とし、必ずUIで修正可能とする。

### 3.9 テクスチャ生成 (FR-10)【Phase 3】
- `texture_mode: none|paint` を追加。`paint` 時は Hunyuan3D-2 paint パイプライン
  (texgen)で全周テクスチャを生成する。
- ビューア用GLBはテクスチャ付きで出力。4色プリント用にはテクスチャ色を
  頂点カラーにサンプリングした上で従来のパレット量子化・色分割3MF出力(FR-8)を適用する。
- texgenのビルド・依存が利用不可の環境では、機能をUI上で無効表示し、
  正面/背面投影方式(FR-8)にフォールバックする。

### 3.10 プリセット (FR-11)【Phase 3】
- パラメータ一括設定のプリセットを提供: `フィギュア`(高さ100mm/384/20万面)、
  `小型フィギュア`(高さ60mm/256/10万面)、`ペンダント`(高さ40mm/256/8万面/カラーOFF)、
  `高精細`(高さ150mm/512/40万面)。選択後も個別調整可能。

### 3.11 サポート必要箇所ヒートマップ (FR-12)【Phase 3】
- ビューアの表示モードに「オーバーハング」を追加。造形方向(Z)に対する面の傾斜角から
  サポートが必要になりやすい箇所(既定45°超)を赤系グラデーションで強調表示する。
  クライアントサイド(Three.js)のみで計算し、閾値スライダーで調整できる。

### 3.12 ぬいぐるみ型紙生成 (FR-13)【Phase 4】

生成済みジョブの3Dメッシュから、ぬいぐるみ縫製用の型紙(2Dパネル)を生成する。

- **モジュール境界(将来の独立リポジトリ化に備える)**: 実装は `server/pattern/`
  に完全に閉じた純粋モジュールとする。入力は `trimesh.Trimesh`(+パラメータ)、
  出力はパネルデータ/SVG文字列のみ。`server/` の他モジュール(generators,
  jobs, config, colorproc等)を**一切importしない**。アプリとの接続は
  main.py / jobs.py 側の薄いアダプタが行う。依存はnumpy/scipy/trimeshのみ
  (平坦化も自前実装とし、追加の外部依存を持たない)。
- **前処理**: 型紙用にメッシュを平滑化+簡略化(目標1万〜2万面。ぬいぐるみは
  細部を布で再現できないため)。
- **パネル分割 (4a)**: 面の法線・曲率ベースのクラスタリングで、指定パネル数
  (4〜12、デフォルト6)に分割。頂点カラーがある場合は色境界をシームに
  誘導するオプション(`use_colors`、デフォルトON)。各パネルは円盤位相
  (穴なし・連結)になるよう後処理する。
- **平坦化 (4b)**: 各パネルをLSCM初期解+ARAP反復で2D展開。パネルごとの
  歪み指標(面積歪み・辺長歪みの最大/平均)を算出し、閾値超過はレポートする。
- **型紙出力 (4b)**: 実寸SVG(mm単位、A4/A3タイル印刷を想定した実寸viewBox)。
  縫い代(`seam_allowance_mm`、デフォルト7mm)オフセット線、パネル番号、
  縫い合わせ対応を示す合印(シーム両側で同一記号)、布目線(グレインライン)を含む。
- **API**: `POST /api/jobs/{id}/pattern`(params: `n_panels`, `seam_allowance_mm`,
  `use_colors`, `smooth_iterations`)→ 同期実行(数秒)で
  `pattern.svg` / `pattern.json`(パネル統計)/ `pattern_preview.glb`
  (パネル色分けメッシュ)をジョブディレクトリに保存。
  `GET /api/jobs/{id}/pattern.svg` / `pattern.json` / `pattern_preview.glb` で取得。
- **UI**: 生成済みジョブに対する「型紙を生成」パネル(パネル数・縫い代の設定)。
  3Dビューアにパネル色分け表示、SVGダウンロードボタン。
- **2段階構成 (4c)**: 実際のぬいぐるみは部位(頭・胴・腕・脚・耳等)を別々に
  縫製・詰め物して後で縫い付ける構造のため、型紙生成は2段階とする:
  1. **パーツ分解**: メッシュを部位単位のボリュームに分割する。局所肉厚
     (Shape Diameter Function 的な内向きレイキャスト)と凹み(くびれ)を
     手がかりに自動分解し(パーツ数はヒントとして指定可、`n_parts`)、
     ビューアでパーツ色分けを確認・パーツ数を変えて再実行できる。
     切断面は蓋(キャップ)して各パーツを閉じたメッシュにする。
  2. **パーツ単位の型紙作成**: 各パーツに対してパネル分割(パーツごとに
     2〜4枚目安、自動)→平坦化→SVG出力。SVGはパーツごとにグループ化し、
     パーツ名ラベル(部位1, 2, ...とビューアの色の対応凡例)を付す。
     切断面(取付口)は「取付口/詰め口」として太線+ラベルで明示する。
  - **パーツ分解の誘導方式**: `POST .../pattern/parts` に `seeds`(3Dビューア上で
    クリックした2〜48点、`guidance="manual"`)が指定されればそれを最優先で使う。
    **同じ名前のシードは同一パーツに統合される**(マルチソース最短距離。
    「胴体」を正面と背面に1点ずつ打つと、正面シードのみでは波面の出会う位置が
    不適切になりがちな背面の割当が改善する)。ユニークな名前の数(=パーツ数)は
    2〜20(プレビューパレットの20色に対応)。
    未指定の場合は `use_llm`(マルチモーダルLLMのbbox誘導、`guidance="llm"`)→
    `use_image`(入力画像の色領域誘導、`guidance="color"`)→ジオメトリのみ
    (`guidance="geometry"`)の順にフォールバックする。手動シード誘導は
    測地距離ベース(面双対グラフ上の重み付きマルチソースDijkstra)でシード
    最寄りに各面を割り当てる方式で、エッジ重みは凹み(くびれ)ペナルティと
    肉厚遷移ペナルティ(|Δlog(局所肉厚)|、スケール不変)の**加算形**
    (距離 × (1 + w_凹み×凹み強度 + w_肉厚×|Δlog肉厚|))。肉厚項により、
    生成メッシュの背面(画像補完由来で凹みの溝が浅い)や耳の付け根のように
    凹み信号が弱い箇所でも、肉厚コントラスト(耳=薄い/頭=厚い)で境界が
    部位の付け根へ誘導される。乗算形は両ペナルティの交差項により、凹みと
    肉厚急変が共存する箇所(脇の下・腕の付け根)のコストが過大になり
    境界が溝を避けて乱れるため採用しない。
  - **背面への仮想シード自動伝播**(`propagate_back`、既定true): 頭や胴体は
    丸/円柱系の形状のため、正面のシードから面の逆法線方向(メッシュ内部)へ
    レイを飛ばして最初に外へ抜ける交点(first exit)は高確率で同じパーツの
    背面スキンである。そこへ同グループの**仮想シード**を自動生成し、正面に
    しかシードを打っていない場合の背面の割当を改善する。レイの向きには
    シード面の単一法線ではなく近傍の面積重み平均法線を使う(実メッシュの
    局所的なバンプで法線が傾き、レイが隣のパーツまで貫通するのを防ぐ)。
    さらに、平均法線を中心とした広角コーン(60°、20本)で対蹠アライメント
    探索を行い、「レイ方向とexit面法線の内積(対蹠アライメントスコア)」が
    最大の候補を採用する(単一の逆法線方向だけだと、シードが部位の縁寄り
    (頭頂近くなど)にある場合、幾何学的な直近対蹠点が隣接パーツ側に落ちて
    しまうため)。誤配置ガードとしてトンネル検出(exitまでの距離が局所肉厚
    推定の3倍を超える候補を除外してから対蹠アライメントスコアで選ぶ)・
    所有権チェック(exit面の予備Dijkstra割当で自グループから測地的に遠すぎる
    候補の棄却)・**ペア単位均衡**(グループGの候補は、Gと隣接する各グループ
    Hが候補を持つかHがコンパクト=面積比6%以下の小パーツの場合のみ採用。
    片側の大パーツだけの背面強化による境界の偏りを防ぐ)を適用する。仮想
    シードはDijkstraのソースに加わるが多数決平滑化ではピン固定されない
    (ユーザーの明示シードとの信頼度差)。生成された仮想
    シードは結果の `n_virtual_seeds`/`virtual_seeds` に、得られなかった
    グループの棄却理由(no_hit/tunnel/taken/ownership/balance)は
    `virtual_seed_skips` に記録される。
    **既知の限界**: 帽子のような小パーツが厚い部位(頭)に隣接する場合、
    `compute_local_thickness` の拡散補間・平滑化により小パーツ側の局所
    肉厚が過大評価され、トンネル検出の閾値が緩くなって「頭を貫通して
    首元まで抜ける誤exit」がプールに残り、対蹠アライメントスコアで正しい
    候補より高スコアになって採用されてしまうケースがある。この誤exitの
    所有権比率は、十字状に交差する部位のような「正しい対蹠面が測地的に
    遠い」正当ケースと値域が重なるため、閾値調整や複合スコアリングでの
    分離は困難(検証の詳細は `server/pattern/parts.py` の
    `_opposite_virtual_seed_faces` docstring参照)。この場合はペア単位
    均衡・多数決平滑化による部分的な緩和(誤って独占された背面領域の
    一部を隣接パーツが取り戻す)に委ねる。
  - **境界の平面フィット正則化**(`planar_boundaries`、既定true): 隣接パーツ
    ペアの境界エッジ中点群に、信頼度重み(Dijkstraのエッジペナルティと同じ
    w_凹み×凹み強度+w_肉厚×|Δlog肉厚|)付きのPCAで平面をフィットし、
    ペアの全面を平面のどちら側かで切り直す(フルカット。「正面の脇の深い溝で
    検出した区切りを肩・背面へそのまま伸ばす」ユーザー提案の一般化。境界が
    平面的な楕円=縫いやすい取付口に整う)。フィットは**アンカーベースの
    トリム反復**でロバスト化: 信頼度が中央値の3倍以上の境界点(実際に
    コントラストがある場合のみ)から開始し、残差の大きい点をトリムして
    再フィットを繰り返すことで、染み出してうねった低信頼区間を外れ値として
    排除する。ガード: 境界エッジ数/アンカー数不足/非平面(アンカー共分散の
    固有値比λ_min/λ_mid過大=鞍状)/カバレッジ不足(アンカーが境界全体の
    一部の弧に集中)/確信面が平面を分離しない/シード面が平面の誤った側/
    再割当の規模異常、のいずれかでペア単位スキップ(ペアごとの適用可否・
    理由・rms・anchor_rms・n_anchors は結果の `planar_fit` に記録)。
    1ペアも適用されない場合は結果を一切変更しない。
- **制約**: 詰め物による膨張の逆補正は簡易(一律スケール係数)に留める。
  数学的な可展面保証はせず、布の伸縮で吸収可能な歪み(目安: 辺長歪み±10%以内)を
  目標とする。パーツ自動分解の失敗時はユーザーがパーツ数を変えて再実行する
  運用とし、対話的な切断編集は将来拡張。シーム直線化・縫い合わせ長の厳密整合・
  膨張補正はPhase 4dで扱う。

### 3.13 ジョブ履歴 (FR-7)
- セッション中に生成したジョブの一覧(サムネイル・状態・作成日時)を表示し、
  過去の生成結果をビューアで再表示・再ダウンロードできる。
- ジョブデータはサーバ側 `data/jobs/<job_id>/` に永続化(入力画像・メッシュ・メタデータJSON)。

## 4. 非機能要件

| ID | 要件 |
|---|---|
| NFR-1 | 生成時間: Hunyuan3D-2使用時、1モデルあたり60秒以内を目標(本GPU環境) |
| NFR-2 | 同時実行: 生成ジョブはGPUメモリ保護のため直列実行(キュー処理)。 |
| NFR-3 | モデルのロードは初回リクエスト時に1度だけ行い、以降常駐させる。 |
| NFR-4 | ローカル利用前提のため認証は不要。バインドは127.0.0.1をデフォルトとする。 |
| NFR-5 | mockジェネレータによりGPU/モデル無しでも全API・UIがE2Eで動作すること。 |
| NFR-6 | VRAM最小要件(既定パラメータでの実測ピーク+余裕): 形状生成のみ 16GB(実測ピーク約12GB)、テクスチャ生成(FR-10)併用時 32GB(実測ピーク約25GB)。mockはGPU不要。詳細はREADME「VRAM最小要件」参照。 |

## 5. API仕様(REST)

ベースURL: `http://127.0.0.1:8100`

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/jobs` | multipart: `image`(必須), `params`(JSON文字列, 任意)。ジョブ作成→ `{job_id}` |
| GET | `/api/jobs` | ジョブ一覧 |
| GET | `/api/jobs/{id}` | ジョブ状態・メタデータ(進捗、メッシュ統計含む) |
| GET | `/api/jobs/{id}/input` | 入力画像(背景除去後があればそれ) |
| GET | `/api/jobs/{id}/model.glb` | ビューア表示用GLB |
| GET | `/api/jobs/{id}/download?format=stl\|3mf\|obj\|glb` | エクスポート |
| DELETE | `/api/jobs/{id}` | ジョブ削除 |
| GET | `/api/health` | ヘルスチェック(使用ジェネレータ名・GPU情報含む) |
| GET | `/` | フロントエンドUI |

## 6. 画面仕様

シングルページ構成(左右2ペイン):
- **左ペイン**: 画像アップロード(D&D)、パラメータ設定フォーム、生成ボタン、
  進捗表示、ジョブ履歴リスト。
- **右ペイン**: Three.js 3Dビューア(全高)、上部にモデル情報バー、
  下部にエクスポートボタン群(STL / 3MF / GLB / OBJ)。
- 日本語UI。

## 7. 制約・前提

- キャラクターシートからの厳密なマルチビュー再構成(正面+側面+背面の整合)は
  本バージョンのスコープ外。単一画像からの生成品質で対応する。
- テクスチャ(カラー3Dプリント)は将来拡張(Hunyuan3D-2 paint pipeline)。
- サポート材生成・スライスはスライサー(Cura等)の役割とし、本アプリは行わない。
